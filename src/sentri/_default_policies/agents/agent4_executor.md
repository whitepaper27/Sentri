# Agent 4: The Executor (Safe Runner)

## Purpose

Execute database fixes with comprehensive safety checks, atomic transactions,
automatic rollback on failure, and immutable audit logging. The Executor is the
only agent with write privileges on target databases. Every action it takes is
guarded by pre-conditions, validated by post-conditions, and recorded in the
audit trail.

The Executor embodies the principle of "safe by default" -- it will refuse to act
if any safety condition is not met, preferring to escalate to a human rather than
risk an unsafe change.

## Input

| Parameter | Description |
|-----------|-------------|
| **ExecutionPlan** | Generated by the orchestrator, contains forward SQL, rollback SQL, validation queries, and metadata |
| **Workflow** | The full workflow record from SQLite, including Suggestion and VerificationReport |
| **Environment** | Target database environment (DEV, UAT, PROD) determining approval requirements |

The ExecutionPlan structure:

```
{
  "workflow_id": "wf-20260212-001",
  "database_id": "PROD-DB-07",
  "environment": "PROD",
  "forward_sql": ["ALTER TABLESPACE USERS ADD DATAFILE SIZE 10G AUTOEXTEND ON"],
  "rollback_sql": ["ALTER TABLESPACE USERS DROP DATAFILE '/path/to/new/file'"],
  "validation_query": "SELECT used_percent FROM dba_tablespace_usage_metrics WHERE tablespace_name = 'USERS'",
  "validation_threshold": {"used_percent": {"operator": "<", "value": 85}},
  "risk_level": "LOW",
  "estimated_duration_seconds": 12
}
```

## Output

An **ExecutionResult** object stored as JSON in the `execution_result` column of the
`workflows` table. Contains:

- `status` -- `SUCCESS`, `FAILED`, `ROLLED_BACK`, `ESCALATED`
- `forward_result` -- Output/result of the forward SQL execution
- `validation_result` -- Output of the post-execution validation query
- `rollback_result` -- Output of rollback if it was triggered (null otherwise)
- `duration_seconds` -- Wall-clock time for the full execution cycle
- `audit_id` -- Reference to the immutable audit_log record

## Execution Steps

The Executor follows these steps in strict order. If any step fails, execution
halts and the appropriate error handling path is taken.

### Step 1: Acquire Resource Lock

Request an exclusive lock on the target resource to prevent concurrent modifications.
The lock key follows the format: `{database_id}:{resource_type}:{resource_name}`.

Example: `PROD-DB-07:TABLESPACE:USERS`

If the lock cannot be acquired within the timeout, the execution is deferred and
retried after the lock is released.

### Step 2: Validate Rollback Plan Exists

Verify that the ExecutionPlan contains a non-empty `rollback_sql` array. If no
rollback plan is defined, the Executor **refuses to proceed** and escalates to
a human DBA. This is a hard safety requirement with no override.

### Step 3: Check Environment Approval Status

Verify that the workflow has the appropriate approval for its environment:

| Environment | Requirement |
|-------------|-------------|
| **DEV** | No approval needed -- auto-execute |
| **UAT** | Approval required for high-risk changes; low-risk auto-executes |
| **PROD** | Approval **always** required -- verify `approved_by` is set |

If approval is required but not present, halt execution and return the workflow
to `AWAITING_APPROVAL` status.

### Step 4: Execute Forward SQL

Connect to the target database with privileged credentials and execute the
forward SQL statements in order. Each statement is executed within a transaction.

- Capture all output, row counts, and timing information.
- Monitor execution progress for long-running operations.
- If any statement fails, immediately proceed to auto-rollback (Step 7).

### Step 5: Capture Output

Record the complete execution output including:

- SQL statements executed (with bind variable values)
- Row counts affected
- Any Oracle messages or warnings (ORA- codes)
- Wall-clock duration per statement
- Database session ID used

### Step 6: Run Post-Execution Validation

Execute the `validation_query` from the ExecutionPlan and compare the result
against `validation_threshold`. This confirms the fix actually worked.

| Validation Result | Action |
|-------------------|--------|
| Passes threshold | Proceed to commit and mark SUCCESS |
| Fails threshold | Trigger auto-rollback (Step 7) |
| Query error | Trigger auto-rollback (Step 7) |

### Step 7: Auto-Rollback on Failure

If execution or validation fails, execute the `rollback_sql` statements to
restore the database to its pre-change state.

- Execute rollback SQL in reverse order.
- Validate that rollback succeeded by re-running the original verification query.
- If rollback succeeds: mark workflow as `ROLLED_BACK`.
- If rollback fails: mark workflow as `ESCALATED` (see Error Handling).

### Step 8: Release Lock

Release the resource lock acquired in Step 1, regardless of whether execution
succeeded or failed. Locks must never be held indefinitely.

### Step 9: Write Audit Record

Insert an immutable record into the `audit_log` table containing:

- Full SQL executed (forward and rollback if applicable)
- Who executed it (`agent4` or `human_override`)
- Who approved it (for PROD changes)
- Result (`SUCCESS`, `FAILED`, `ROLLED_BACK`)
- Evidence (before/after metrics as JSON)
- Change ticket reference (JIRA number if applicable)

This record is append-only and must never be modified or deleted.

## Safety Rules

1. **Never execute without a rollback plan.** If `rollback_sql` is empty or null,
   the Executor refuses to proceed. No exceptions. No override flag.

2. **Atomic transactions only.** All forward SQL runs in a single transaction.
   Either all statements succeed and are committed, or none are.

3. **Always validate after execution.** The validation query must confirm the fix
   worked. A "successful" SQL execution that does not improve the metrics is
   treated as a failure.

4. **Respect environment boundaries.** Never execute on a database in a higher
   environment than approved. A DEV-approved plan cannot run on PROD.

5. **One change at a time.** The resource lock ensures only one fix is applied
   to a given resource at any time. No concurrent modifications.

6. **Privilege separation.** The Executor uses dedicated database credentials
   that are separate from the Auditor's read-only credentials.

## Timeout

| Operation | Timeout | Action on Timeout |
|-----------|---------|-------------------|
| Lock acquisition | **30 seconds** | Defer execution, retry after lock holder releases |
| SQL execution (total) | **300 seconds** (5 minutes) | Abort execution, attempt rollback |
| Individual SQL statement | **120 seconds** | Abort statement, attempt rollback |
| Validation query | **30 seconds** | Log warning, treat as validation failure, rollback |
| Rollback execution | **120 seconds** | Mark as ESCALATED, page DBA immediately |

## Error Handling

- **Execution failure** (ORA- error during forward SQL): Immediately execute the
  rollback SQL. Log the full error including ORA- code, SQL statement, and bind
  variables. Mark workflow as `ROLLED_BACK` if rollback succeeds.

- **Rollback failure**: This is a **critical event**. Mark the workflow as
  `ESCALATED`. Send immediate notification via all channels (Slack, PagerDuty,
  email). Halt all further automated operations on the affected database until
  a human DBA intervenes. Log at CRITICAL level.

- **Lock contention**: If the lock cannot be acquired, do not queue indefinitely.
  Wait up to 30 seconds, then defer the execution. The orchestrator will retry
  on the next poll cycle.

- **Connection loss during execution**: If the database connection drops mid-
  execution, attempt to reconnect and check the state of the transaction. If
  the transaction was committed, validate. If not, the database will have
  auto-rolled back.

- **Timeout during execution**: Attempt to cancel the running SQL statement,
  then execute rollback. If cancellation fails, escalate immediately.
